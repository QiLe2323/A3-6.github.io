Development Process

Development Process

The development process of Museum Tales followed an iterative path from concept design to a working Web prototype. We did not build the final system directly at the beginning. Instead, we first tested whether the core interaction loop could work, then gradually improved the interface, interaction logic, visual design, and technical implementation.

Stage 1: Concept Development

The first stage focused on defining the project direction. Our initial goal was to use AR and interactive storytelling to make the museum visit more engaging. We wanted users to do more than simply read exhibit descriptions. Therefore, we developed the idea of letting users choose a role, scan exhibits, collect clues, interact with NPCs, and reconstruct a story.

At this stage, we mainly focused on the experience logic. We asked questions such as: What does the user do first? How does scanning connect to the story? Why would users talk to NPCs? How can the ending feel meaningful? These questions helped us define the basic structure of the system before moving into interface and technical development.

Stage 2: Early Functional Prototype

The second stage was to create an early functional prototype. This version was mainly used to test whether the full gameplay loop could run. At this stage, some functions were simplified. For example, instead of real AR recognition, users could input simulated exhibit codes such as E1 to E9. These codes represented different exhibits and triggered corresponding clues or story fragments.

Scanning Method Decision

Scanning Method Decision

During the early prototype stage, we also compared different scanning approaches. This helped us decide how to balance technical feasibility, stability, and immersion in the current prototype.

Scanning Interaction Decision.

During the implementation stage, we also considered the difference between QR code scanning and AR-based physical exhibit recognition. In this project, scanning is not designed as a simple QR code interaction. Although QR codes are stable and easy to implement, they create a weaker sense of immersion because users interact with an external code rather than the museum object itself.

Museum Tales instead uses AR-based recognition of physical exhibits or cultural postcards. This means that users scan the visual appearance of the exhibit or postcard itself, and the system then unlocks related clues, NPC dialogue, and story content. This approach is more technically demanding, but it better supports the project’s goal of transforming museum objects into story triggers.

Scanning Method Strengths Limitations
QR Code Scanning Stable, easy to implement, accurate in recognition, and commonly used for quickly linking users to digital content Less immersive, because users scan an external code rather than interacting directly with the exhibit or postcard itself
AR Physical Exhibit / Postcard Recognition More natural and closer to a real museum AR experience, because users scan the visual appearance of the exhibit or cultural postcard itself More technically demanding, as it depends on lighting, camera angle, image quality, visual features, and recognition stability
Museum Tales Approach Uses AR recognition to connect physical exhibits or cultural postcards with story clues, NPC dialogue, and narrative progression Requires careful preparation of recognisable exhibit/postcard images and clear scanning feedback so users understand when content has been unlocked

The early prototype included identity selection, simulated scanning, NPC dialogue, clue collection, ending conditions, and a basic result page. Although the interface was still simple and text-heavy, it was useful because it allowed us to check the basic logic of the system. We could see whether users could move from role selection to scanning, from scanning to NPC interaction, and finally to story reconstruction and ending.

This stage was important because it showed that the project should not start with visual polishing only. The core system logic needed to work first.

Stage 3: Figma Iteration

After validating the basic flow, we moved into Figma design iteration. The interface evolved from low-fidelity wireframes to mid-fidelity and high-fidelity designs. During this process, we refined the role selection page, Mission Hub, scanning page, NPC page, Profile page, and Ending page.

The focus of this stage was to improve clarity and immersion. Early screens were too text-heavy, so we redesigned the interface with clearer hierarchy, cards, tabs, buttons, and visual cues. We also changed the visual direction from a more technical black-and-neon style to a softer cream, light-gray, and muted green design. This made the interface feel more suitable for a museum storytelling experience.

Figma helped us test layout and visual structure before implementation. It also allowed us to compare different UI options and decide which elements should be kept in the final Web version.

Stage 4: Web UI Development

The fourth stage was to translate the Figma design into a working web interface. We used Next.js, React, TypeScript, and Tailwind CSS to build the main pages and interactive components.

The Web UI included the Welcome page, Role Selection page, Mission Hub, Scan Exhibits page, NPC Conversation page, Story Reconstruction page, Ending page, and Profile / Ranking page. Each page was built as part of the complete user journey rather than as an isolated screen.

During this stage, we paid attention to both visual consistency and interaction usability. Components such as role cards, clue cards, NPC interaction panels, navigation bars, and result cards were designed to be reusable. This helped keep the interface consistent and made the development process more manageable.

Stage 5: Interaction Logic Development

After the main pages were built, we implemented the interaction logic behind the interface. This included role state, scanned exhibit state, clue collection, NPC interaction conditions, story reconstruction logic, ending generation, and ranking records.

For example, when a user selects a role, the system stores that identity and uses it to shape later tasks and narrative context. When the user scans an exhibit or postcard, the system unlocks related clue keywords. These clues then affect what the user can understand or use during NPC dialogue and the final story reconstruction stage.

The ending system uses the user’s progress and story completion state to generate a result, including ending title, score, level, and unlocked clue summary. The profile and ranking system then records this result and displays it as part of the user’s achievement.

This stage was challenging because the experience needed to feel like one connected story, not a collection of separate pages. Therefore, the logic between role, scan, clue, NPC, reconstruction, and ending had to be carefully connected.

Stage 6: Refinement

The final stage focused on refinement. After building the prototype, we reviewed the user flow, page clarity, interaction feedback, and visual hierarchy. Some parts of the system were adjusted because they were too text-heavy or not clear enough for first-time users.

We improved the Mission Hub so that users could better understand their current tasks and progress. We made clue feedback more visible after scanning. We also refined the NPC page so that characters felt more like part of the story rather than simple buttons. The final visual style was adjusted to be softer, cleaner, and more consistent with the museum context.

This stage helped the prototype become more complete and usable. It also made the final result feel closer to an actual interactive museum experience.
