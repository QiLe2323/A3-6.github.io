AI Use

Technical Reflection on AI Use: Vibe Coding

During the development of Museum Tales, we used AI-assisted coding, or “vibe coding”, as a supportive method to speed up prototyping and explore possible technical solutions. Instead of writing every line of code completely from scratch, we sometimes described the feature we wanted in natural language and asked AI to generate a first version of the code. However, the AI-generated output was never used directly without checking. We still needed to test, debug, modify, and connect the code with our own project structure.

Overall, AI was useful for generating initial implementation ideas, reusable component structures, and logic examples. However, it also produced code that was sometimes too generic, not fully connected to our existing system, or missing important edge cases. Therefore, our process was not simply “ask AI and copy the answer”. It was closer to a cycle of prompting, testing, correcting, and integrating.

Example 1: Role Selection and State Management

P — Prompt

We asked AI to help us implement the role selection logic for the Museum Tales Web prototype. The context we provided was that users should choose one role before entering the Mission Hub, and the selected role should affect later pages such as scanning, NPC dialogue, clue collection, and ending generation.

A simplified version of the prompt was:

Create a React / Next.js role selection component for a Web AR museum storytelling app. Users should choose one of four roles. After clicking “Start Journey”, the selected role should be saved and used in later pages such as Mission Hub, Scan Exhibits, NPC Conversation, and Ending.

O — Outcome

The AI generated a basic role selection component with four role cards and a button to continue to the next page. This was useful because it gave us a quick starting point for the layout and helped us think about how to structure role data, such as role name, description, icon, and route.

However, the first version was too isolated. It only stored the selected role inside the component state, which meant the role information could be lost when users moved to another page or refreshed the browser. It also did not match our actual project structure, because our system needed the role state to be shared across multiple pages.

So the AI output was useful as a first draft, but it was not complete enough for the real prototype.

A — Action

We modified the code by connecting the role selection to our global state management. Instead of keeping the role only inside one component, we used a shared state structure so that the selected role could be accessed by the Mission Hub, scanning page, NPC dialogue, and ending logic.

We also adjusted the role data manually to fit our narrative design. The AI-generated role descriptions were too generic, so we rewrote them to better match the Museum Tales experience and the Xi’an Jiaotong-Liverpool University Museum context. After testing, we refined the button states, selected-card feedback, and navigation logic to make the role selection feel more stable and meaningful.

This example showed that AI was helpful for quickly generating a basic interface structure, but the important design logic still needed to be manually adapted to our own system.

Example 2: Clue Collection and Story Progression Logic

P — Prompt

We also used AI to help us explore the logic between scanning exhibits, collecting clues, talking to NPCs, and unlocking the story reconstruction stage. The context we provided was that each exhibit or postcard should unlock specific clue keywords, and these clues should later influence NPC dialogue and the final story reconstruction.

A simplified version of the prompt was:

Help me design JavaScript / TypeScript logic for a museum storytelling web app. When users scan an exhibit, the system should unlock related clue keywords. The collected clues should be stored, shown in the Mission Hub, and used to determine whether users can enter the story reconstruction stage. The logic should also support different roles and endings.

O — Outcome

The AI produced a basic object-based logic structure. It suggested storing exhibit IDs, clue keywords, role conditions, and ending conditions in arrays or objects. This was helpful because it gave us a clearer way to organise the relationship between exhibits, clues, NPCs, and endings.

However, the output had some problems. Some conditions were too simple and did not fully reflect our story design. For example, the AI treated clue collection as a simple checklist, but our project needed clues to feel more narrative and connected to NPC dialogue. The first version also did not clearly handle cases where users scanned the same exhibit multiple times, collected duplicate clues, or tried to enter story reconstruction too early.

So the AI output was partially successful, but it needed significant revision before it could be used.

A — Action

We revised the generated logic and made it more suitable for our actual prototype. We added clearer clue states, such as whether a clue was newly unlocked, already collected, or required for story reconstruction. We also adjusted the conditions for entering the reconstruction stage so that users needed to collect enough relevant clues before moving forward.

We manually connected the clue logic with the Mission Hub and NPC dialogue pages. This allowed the Mission Hub to show user progress more clearly, while the NPC page could respond differently depending on which clues had already been collected. We also refined the ending logic so that the final result was based on user progress rather than simply reaching the last page.

This example showed that AI was useful for generating a logical framework, but it could not fully understand the narrative meaning of each clue. We still had to make design decisions ourselves and adjust the code according to the intended user experience.

Overall Reflection

Using AI for vibe coding helped us move faster during the early development stage. It was especially useful when we needed a quick starting point for React components, state logic, data structures, or interaction flows. It also helped us explore different implementation possibilities before deciding which one fit our prototype best.

However, the AI-generated code was not always reliable or project-specific. Sometimes it produced generic code that worked in isolation but did not connect well with our existing pages. Sometimes it ignored edge cases, such as page refresh, repeated scanning, unclear user progress, or incomplete clue conditions. Because of this, we had to carefully test and revise the AI-generated code instead of using it directly.

The most important lesson is that AI can support prototyping, but it cannot replace design judgement. In Museum Tales, the technical implementation was closely connected to user experience and storytelling. Therefore, we needed to decide how each function supported the museum narrative, not just whether the code could run. Vibe coding helped us build faster, but the final system still depended on human testing, debugging, narrative design, and interface refinement.
